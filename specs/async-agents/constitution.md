# Constitution

## Project Purpose

**Async/Hybrid Workflow Collaboration for ai-sdd** — Extend the ai-sdd framework to support a mix of synchronous and asynchronous tasks within workflows, coordinated through collaboration tools (Slack, Confluence, Jira, Bitbucket, GitHub).

### Vision
Currently ai-sdd runs workflows synchronously — each agent executes inline and the engine blocks until completion. This feature introduces **async task execution** where agents perform work, notify stakeholders via Slack, and the orchestrator waits for approval signals before advancing the state machine. Workflows can be fully sync, fully async, or hybrid (mixed).

### Core Capabilities
- **Async task mode**: tasks can be configured as `sync` or `async`; the engine handles both execution models
- **State machine**: orchestrator manages async lifecycle (AWAITING_APPROVAL → APPROVED → DOING → DONE)
- **Slack as primary coordination bus**: agents notify and listen via Slack channels
- **Confluence integration**: document authoring, inline/standard commenting, review feedback
- **Configurable stakeholder sign-off**: minimum approval threshold before state transitions
- **PR-based code review flow**: raise PR → Slack notify → review feedback → address → merge
- **Jira-as-Code**: workflow YAML / task specs are the single source of truth for Jira tickets (epics, stories, sub-tasks). Jira is managed via code, not the UI. Task creation, state transitions, and metadata sync bidirectionally but code always wins on conflict. Same model for GitHub Issues/Projects in MVP 2.

### MVP 1 — Atlassian Stack
- Slack (notifications, inter-agent coordination)
- Confluence (document authoring & review)
- Jira (task tracking, Agile Kanban workflow, epics/sub-tasks)
- Bitbucket (PRs, code review, pipelines)

### MVP 2 — GitHub Stack
- GitHub abstraction mimicking Jira via Issues + Projects + Labels
- GitHub PRs and code review
- GitHub Actions (CI/CD)
- Same Slack + Confluence integrations

### Post-MVP (future)
- Auto-triggered deployment pipelines (CI → SIT → canary → prod)
- Metric collection and automated smoke tests
- Advanced approval routing (role-based, time-boxed)

## Core Domain Entities

| Entity | Description |
|---|---|
| `AsyncTask` | A workflow task configured for async execution with approval gates |
| `ApprovalSignal` | A stakeholder sign-off event (approve/reject with optional notes) |
| `CollaborationChannel` | A configured Slack channel for agent ↔ stakeholder communication |
| `DocumentReview` | A Confluence page + comment thread used for spec review cycles |
| `PullRequestReview` | A Bitbucket/GitHub PR with review feedback loop |
| `TaskBoard` | A Jira board or GitHub Project tracking task states |
| `TaskSync` | Bidirectional sync between workflow YAML task definitions and Jira/GitHub tickets (code is source of truth) |
| `StateTransition` | An orchestrator-managed state change with audit trail |

## Standards

### Quality
- All spec documents must include a `## Summary` section
- Architecture docs must include a `## Components` section with interface contracts
- Collaboration tool integrations must be behind well-defined adapter interfaces (pluggable)
- State machine transitions must be auditable (event log with timestamps, actor, transition)

### Technology Constraints
- **Runtime**: Bun + TypeScript strict mode (consistent with ai-sdd core)
- **No vendor lock-in**: each collaboration tool behind an adapter interface — swappable
- **Slack integration**: keep it simple for now — channel posting + message listening, minimal interactive features
- **Auth**: all tool credentials via environment variables or secure config, never hardcoded
- **Jira-as-Code**: code (workflow YAML + task specs) is the single source of truth — Jira/GitHub tickets are derived artifacts, never manually edited

### Review Criteria
- BA sign-off required before architecture proceeds (HIL gate on `define-requirements`)
- L1 architecture must cover: async engine extensions, state machine, adapter layer, Slack bus, and approval flow
- MVP 1 and MVP 2 must share the same adapter interfaces — GitHub adapters implement the same contracts as Atlassian ones
- State machine design must handle: timeouts, rejection loops, concurrent approvals, and partial failures
